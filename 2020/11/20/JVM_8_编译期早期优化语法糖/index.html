先谈一下编译

在Java技术下谈“编译期”而没有具体上下文语境的话，其实是一句很含糊的表述，因为它可能是指一个前端编译器（叫“编译器的前端”更准确一些）把 .java文件转变成.class文件的过程；也可能是指Java虚拟机的即时编译器（常称JIT编译器，Just In Time Compiler）运行期把字节码转变成本地机器码的过程；还可能是指使用静态的提前编译器（常称AOT编译器，Ahead Of Time Compiler）直接把程序编译成与目标机器指令。

我们接下来将要讨论的是前端编译器的优化，也就是javac编译器，而后期优化将会在稍后讨论。

语法糖

所谓的 语法糖，其实就是指 javac 编译器把 .java 源码编译为 .class 字节码的过程中，自动生成和转换的一些代码，主要是为了减轻程序员的负担，算是 java 编译器给我们的一个额外福利（给糖吃嘛），常见的语法糖有以下几种。

默认构造

  • 当新建一个类（继承自Object）并且没有为它写构造器时，编译器会自动为这个类加上一个无参的构造器，这个无参的构造器内部会默认调用Object的构造方法。
  • 当新建一个类（继承自其他类）并且没有为它写构造器时，如果父类有无参构造，那么编译器也会自动为这个类加上一个无参的构造器，这个无参的构造器内部会默认调用父类的构造方法；如果父类只有有参构造方法，这时必须定义自己的构造方法并调用父类的构造方法，否则会报错，因为编译器只会在你没有写构造方法时自动帮你加上无参构造方法。

可以看到，执行了两遍 switch，第一遍是根据字符串的 hashCode 和 equals 将字符串的转换为相应 byte 类型，第二遍才是利用 byte 执行进行比较。为什么第一遍时必须既比较 hashCode，又利用 equals 比较呢？hashCode 是为了提高效率，减少可能的比较；而 equals 是为了防止 hashCode 冲突。

顺便提一下 hashcode() 和 equals() 的设计原则：

对象相同，hashcode 一定相同（等价表述：hashcode不相同，对象一定不相同）；hashcode 相同，对象不一定相同；所以在 HashMap 中作为 key 的对象一定要重写 equals 和 hashcode 方法，这样设计一是为了提高效率，在匹配桶位时，先通过 hashcode 进行散列，相同对象的 hashcode 肯定散列到同一桶位；二是为了防止 hash 冲突，因为可能存在不同对象产生相同 hashcode 的情况，然后再进入链表 or 红黑树通过 equals 方法判断是否存在相同对象，存在则更新，不存在则插入。
