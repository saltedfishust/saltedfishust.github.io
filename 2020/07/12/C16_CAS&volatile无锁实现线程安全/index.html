CAS & volatile 无锁实现线程安全

发表于 2020-07-12 | 分类于 并发编程
字数统计: 2.7k | 阅读时长 ≈ 10

CAS 简介

前面我们在 初识线程安全问题 中谈到有一种非阻塞同步方式也可以解决线程安全问题，就是 CAS 结合 volatile 实现的无锁并发（非阻塞同步）。

CAS 的含义就是 比较并交换（Compare-and-Swap），这是一项依赖于硬件实现的操作。在 Java 中，有一个很底层的类—— Unsafe，里面提供了几种本地方法用于实现 CAS 操作，不过由于 Unsafe 类在设计上就不是提供给用户程序调用的类 ，因此在 JDK 9 之前只有 Java 类库可以使用 CAS，譬如 JUC 包里面的 AtomicInteger 类，其中的 compareAndSet() 和 getAndIncrement() 等方法都使用了 Unsafe 类的 CAS 操作来实现。而如果用户程序也有使用 CAS 操作的需求，那要么就采用反射手段突破 Unsafe 的访问限制，要么就只能通过 Java 类库 API 来 间接使用它。直到JDK 9之后，Java 类库才在 VarHandle 类里开放了面向用户程序使用的 CAS 操作。

CAS指令需要有三个操作数，分别是内存位置（在Java中可以简单地理解为变量的内存地址，用 V 表示）、旧的预期值（用 A 表示）和准备设置的新值（用 B 表示）。CAS 指令执行时，当且仅当 V 符合 A 时，处理器才会用 B 更新 V 的值，否则它就不执行更新。但是，不管是否更新了 V 的值，都会返回 V 的旧值，上述的处理过程是一个原子操作，执行期间不会被其他线程中断。

CAS 的特点

因为 CAS 往往都是搭配 while循环使用，会一直占用CPU，所以适用于线程数少、多核 CPU 的场景下。

  • CAS 是基于乐观锁的思想：最乐观的估计，不怕别的线程来修改共享变量，就算改了也没关系，我吃亏点再重试呗。
  • synchronized 是基于悲观锁的思想：最悲观的估计，得防着其它线程来修改共享变量，我上了锁你们都别想改，我改完了解开锁，你们才有机会。
  • CAS 体现的是无锁并发、无阻塞并发，因为没有使用 synchronized，所以线程不会陷入阻塞，这是效率提升的因素之一， 但如果竞争激烈，可以想到重试必然频繁发生，反而效率会受影响。

为什么无锁效率高

  • 无锁情况下，即使重试失败，线程始终在高速运行，没有停歇，而 synchronized 会让线程在没有获得锁的时候，发生上下文切换，进入阻塞。
  • 打个比喻，线程就好像高速跑道上的赛车，高速运行时，速度超快，一旦发生上下文切换，就好比赛车要减速、熄火，等被唤醒又得重新打火、启动、加速… 恢复到高速运行，代价比较大
  • 但无锁情况下，因为线程要保持运行，需要额外 CPU 的支持，CPU 在这里就好比高速跑道，没有额外的跑道，线程想高速运行也无从谈起，虽然不会进入阻塞，但由于没有分到时间片，仍然会进入可运行状态，还 是会导致上下文切换。

借助原子类实现

当然 J.U.C 并发包为提供了封装好的原子类。我们无需这么麻烦的用反射的方式实现 cas，主要分为以下几种：

原子整数
  • AtomicBoolean
  • AtomicInteger
  • AtomicLong

就把上面这些类看成Boolean、Integer、Long 即可，只不过是提供了 CAS 的封装。

原子引用
  • AtomicReference
  • AtomicMarkableReference （带版本更改标记）
  • AtomicStampedReference （带版本号控制）

当我们需要保证线程安全的共享变量不是整数时，可以用 AtomicReference 来实现，与原子整数用法类似，只需要把引用的类型通过泛型传进去即可，AtomicMarkableReference 和 AtomicStampedReference 可以同来解决 ABA 问题。

原子数组
  • AtomicIntegerArray
  • AtomicLongArray
  • AtomicReferenceArray

当我们想保护的并不是引用，而是引用里的某个或全部的元素，这个时候就要用到原子数组。

原子字段更新器
  • AtomicIntegerFieldUpdater
  • AtomicLongFieldUpdater
  • AtomicReferenceFieldUpdater // 域 字段

利用字段更新器，可以针对对象的某个域（Field）进行原子操作，只能配合 volatile 修饰的字段使用，否则会出现异常。

原子累加（计算）器
  • LongAdder （每次只能加减1）
  • DoubleAdder
  • LongAccmulator （可以自己定义计算）
  • DoubleAccmulator

专门做累加的，性能比 getAndIncrement() 、getAndAccumulate 等累加方法更好。

LongAdder (DoubleAdder)性能提升的原因很简单，就是在有竞争时，设置多个累加单元，Therad-0 累加 Cell[0]，而 Thread-1 累加 Cell[1]… 最后将结果汇总。这样它们在累加时操作的不同的 Cell 变量，因此减少了 CAS 重试失败，从而提高性能。LongAdder 的源码设计的非常精巧，其中用到了cas 锁，并且巧妙的避免了缓存行的伪共享问题（我们在JVM-4-堆和GC详解中也有介绍），有兴趣的可以去研究一下 LongAdder 的源码。

CAS 的问题（ABA）

且CAS从语义上来说并不是真正完美的，它存在一个逻辑漏洞：如果一个变量 V 初次读取的时候是 A 值，并且在准备赋值的时候检查到它仍然为 A 值，那就能说明它的值没有被其他线程改变过了吗？这是不能的，因为如果在这段期间它的值曾经被改成 B，后来又被改回为 A，那 CAS 操作就会误认为它从来没有被改变过。这个漏洞称为 CAS 操作的“ABA问题”。

如何解决

AtomicMarkableReference 和 AtomicStampedReference 可以同来解决 ABA 问题。

  • AtomicMarkableReference 可以通过控制版本是否更改标记（boolean 值）来保证 CAS 的正确性；
  • AtomicStampedReference 可以通过控制变量值的版本号（int 值）来保证 CAS 的正确性。

不过目前来说这两个类处于相当鸡肋的位置，大部分情况下 ABA 问题不会影响程序并发的正确性，如果需要解决ABA问题，改用传统的互斥同步可能会比原子类更为高效。
