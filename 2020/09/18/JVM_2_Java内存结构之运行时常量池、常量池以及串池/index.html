Java内存结构之运行时常量池、常量池以及串池

运行时常量池

注：上图的“常量池”有误，应该是“运行时常量池”

运行时常量池与常量池：Class文件即字节码文件中除了有类的基本信息（版本、字段、方法、接口等描述信息），类方法定义，还有一项信息是常量池表（Constant Pool Table），也简称为常量池，用于存放编译期生成的各种字面量与符号引用，类方法定义中的虚拟机指令将通过常量池来执行，这部分内容将在类加载后存放到方法区的运行时常量池中。

  • 常量池：就是一张表，虚拟机指令根据这张常量表找到要执行的类名、方法名、参数类型、字面量等信息；
  • 运行时常量池：常量池是*.class 文件中的，当该类被加载，它的常量池信息就会放入运行时常量池，并把里面的符号地址变为真实地址。运行时常量池相对于Class文件常量池的另外一个重要特征是具备动态性，Java语言并不要求常量一定只有编译期才能产生，也就是说，并非预置入Class文件中常量池的内容才能进入方法区运行时常量池，运行期间也可以将新的常量放入池中，这种特性被开发人员利用得比较多的便是String类的intern()方法；
  • 既然运行时常量池是方法区的一部分，自然受到方法区内存的限制，当运行时常量池无法再申请到内存时会抛出OutOfMemoryError异常。

StringTable(串池)

运行时常量池里有一个重要的组成部分——StringTable，也叫作“串池”，如上图所示，在jdk1.7之后串池被放到了堆中，原因是由于字符串被频繁的使用，放在堆中可以方便高效的进行垃圾回收，而在运行时常量池（方法区)，垃圾回收的条件苛刻，不能及时地进行垃圾回收，从而会占用大量的内存。但在逻辑和分析上仍然可以把串池当做运行时常量池的一部分。

StringTable特性 ：

  • 常量池中的字符串仅是符号，第一次用到时才变为对象;
  • 利用串池的机制，来避免重复创建字符串对象；
  • 字符串变量拼接的原理是 StringBuilder（1.8，线程不安全的，所以比StringBuffer速度快）；
  • 在jdk1.8后，串池被放在堆中，所以串池中的对象也跟堆中的对象一样，可以垃圾回收；
  • 字符串常量拼接的原理是编译期优化可以使用 intern 方法，主动将串池中还没有的字符串对象放入串池
    • 1.8及之后 将这个字符串对象尝试放入串池，如果有则并不会放入，如果没有则放入串池， 会把串池中的对象返回（以下不做特殊说明，都指的是在1.8之后的环境中）
    • 1.7及之前 将这个字符串对象尝试放入串池，如果有则并不会放入，如果没有会把此对象复制一份，放入串池，会把串池中的对象返回

例题2：关于 intern()
case1：

在jdk1.8及之后，调用intern()方法将这个字符串对象s尝试放入串池，显然没有则放入串池，最后把放入串池中的对象即s返回；因为s2是返回的串池中的对象，肯定与x相同；s是放入串池的对象，所以s也与x相同。
在jdk1.7及之前，调用intern()方法将这个字符串对象s尝试放入串池，显然没有则复制一份新的对象，把新的对象放入串池，最后把放入串池中的对象即新对象返回；因为s2是返回的串池中的对象，肯定与x相同；因为放入串池的对象是一个复制s的新对象，所以s与x地址不相同。
case2:

因为这里在intern()之前，串池中已经存在字符串对象”ab”，所以并没有放入串池，返回的都是之前串池的对象即x，这在jdk1.6和1.8中是一样的，所以s2==x，s是堆中的对象显然不等于x。

StringTable性能调优
  • StringTable实质上是一个hashtable，可以通过参数-XX:StringTableSize=size设置哈希表的桶个数，默认是60013，一般来说桶个数越大占用的内存越大，但存取速度会越快；
  • 适时调用intern()方法入池会节约内存，避免堆中存在大量意义重复（地址不同）的对象。
