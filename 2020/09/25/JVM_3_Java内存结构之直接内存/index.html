Java内存结构之直接内存

发表于 2020-09-25 更新于 2021-01-26 分类于 JVM
本文字数： 6.8k 阅读时长 ≈ 12 分钟

直接内存

直接内存（Direct Memory）并不是虚拟机运行时数据区的一部分，也不是《Java虚拟机规范》中定义的内存区域，实际上它属于系统内存。但是这部分内存也被频繁地使用，而且也可能导致OutOfMemoryError异常出现。

在JDK 1.4中新加入了NIO（New Input/Output）类，引入了一种基于通道（Channel）与缓冲区（Buffer）的I/O方式，它可以使用Native函数库（本地函数库）直接分配堆外内存，然后通过一个存储在Java堆里面的 DirectByteBuffer对象作为这块内存的引用进行操作。这样能在一些场景中显著提高性能，因为避免了在Java堆和Native堆中来回复制数据。

特点可总结如下：

  • 常见于 NIO 操作时，用于数据缓冲区；
  • 分配回收成本较高，但读写性能高；
  • 不受 JVM 内存回收管理。

原理解析：

一般文件的读写过程：Java并不具备磁盘读写的能力，如果想要进行读写文件，必须借助调用操作系统的函数（本地方法），CPU的运算状态从用户态切换到内核态，切换到内核态后，就可以借助CPU的函数读取磁盘内容，先在操作系统内存中分配一块缓冲区，在系统缓冲区，Java程序是不能运行的，所以会在Java堆内存里分配一块java缓冲区也就是byte[]，先把数据放在系统缓存区再从系统缓存区存到java缓冲区，因此这样数据读取和写入的效率不高。

而直接内存是一块特殊的系统内存区域，可以通过一个存储在Java堆里面的 DirectByteBuffer对象作为这块内存的引用进行操作，即Java代码可以直接访问直接内存区域，少了一次缓冲区的复制操作，所以提高了文件读写的效率。

如何回收（释放），原理是什么？

在windows环境下，debug后通过观察任务管理器可以看到当把byteBuffer对象置为空时会触发垃圾回收，回收不用的直接内存，但不要以为是JVM进行的回收，直接内存是不受JVM内存回收管理的。为了弄清楚直接内存回收的原理，我们需要去查看一下ByteBuffer.allocateDirect(_1Gb)内部是怎么实现的：

我们可以看到DirectByteBuffer(capacity)构造方法内部调用了Unsafe对象进行了直接内存的分配，Unsafe类是很底层的一个类，可以直接对系统的内存进行分配和回收，但似乎没看到在哪里进行了free，

直接内存的回收是借助虚引用（PhantomReference）来实现的。上面的cleaner是一个虚引用，它关联this对象（即byteBuffer），当所关联的对象被垃圾回收时，JVM会收到一个通知从而触发（执行）虚引用对象中的clean()方法，但这个clean()方法并不是在主线程执行的，是在后台的一个守护线程ReferenceHandler执行的，clean()方法运行后会调用传入的任务对象Deallocator进行对直接内存的释放，

真正的回收unsafe.freeMemory(address)出现在任务对象Deallocator里。

所以总结而言直接内存的分配和回收原理如下：

  • 使用了 Unsafe 对象完成直接内存的分配回收，并且回收需要主动调用 freeMemory 方法；
  • ByteBuffer 的实现类内部，使用了 Cleaner （虚引用）来监测 ByteBuffer 对象，一旦 ByteBuffer 对象被垃圾回收，那么就会由 ReferenceHandler 线程通过 Cleaner 的 clean 方法调用 freeMemory 来释放直接内存。
