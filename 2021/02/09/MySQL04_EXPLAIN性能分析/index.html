概念

学完 SELECT 的查询处理，我们再学个更有趣的指令，在 SELECT 语句之前加个 EXPLAIN，模拟优化器执行 SQL 查询语句，可以帮助我们诊断这个查询语句的各项指标，从而知道 MySQL 是如何处理SQL 语句的，分析查询语句或是表结构的性能瓶颈。

用法：EXPLAIN + SQL语句

id （重要）

select 查询的序列号,包含一组数字，表示查询中执行 select 子句或操作表的顺序。

  • id 如果相同，可以认为是一组，从上往下顺序执行；
  • id 如果不同，id 值越大优先级越高，越先被执行；
  • 当指向 UNION_RESULT 时其值为 NULL。

select_type

select_type 属性 含义
SIMPLE 简单的 select 查询,查询中不包含子查询或者 UNION
PRIMARY 查询中若包含任何复杂的子部分，最外层查询则被标记为 Primary
DERIVED 在FROM列表中包含的子查询被标记为 DERIVED(衍生)，MySQL 会递归执行这些子查询, 把结果放在临时表里。
SUBQUERY 在SELECT或WHERE列表中包含了子查询，又称为独立子查询：独立子查询不需要与外部的表交互，所以相比相关子查询更快
DEPEDENT SUBQUERY 在SELECT或WHERE列表中包含了子查询,子查询基于外层，又称为相关子查询：需要与外部的表交互，逻辑IO次数较多，所以相比独立子查询较慢
UNCACHEABLE SUBQUERY 无法使用缓存的子查询
UNION 若第二个SELECT出现在UNION之后，则被标记为UNION；若UNION包含在FROM子句的子查询中,外层SELECT将被标记为：DERIVED
UNION RESULT 从UNION表获取结果的SELECT

SIMPLE

SIMPLE 代表单表查询，无需解释。

PRIMARY

查询中若包含任何复杂的子部分，最外层查询则被标记为 Primary。例如上面 id 那个例子。

DERIVED

派生表（Derived Table），是复杂SQL在执行过程中的中间表，也可认为是临时表，在 FROM 列表中（也可能在join之后）包含的子查询被标记为 DERIVED(衍生)，MySQL 会递归执行这些子查询，把结果放在临时表里。

可以看到第一种写法并没有出现派生表，为什么哪？因为MySQL对这种简单的语句进行了优化，对于简单的派生表，比如没有使用group by，having，distinct，limit等，这类派生表可直接合并到外部的查询块中，上面那个select会被优化成为select * from t1，所以就是一个简单的单表查询，而后面由于我们使用了group by，不能进行合并了所以出现了派生表。

ref: MySQL派生表优化

SUBQUERY

一般指在 SELECT 或 WHERE 列表中包含的子查询，如果是在 FROM 或者 JOIN 之后的那就是 DERIVED了。又称为独立子查询，独立子查询不需要与外部的表交互，所以相比相关子查询更快。

DEPENDENT SUBQUERY

在SELECT或WHERE列表中包含了子查询，子查询基于外层，具体而言是指引用了外部的查询列，又称为相关子查询，需要与外部的表交互，逻辑IO次数较多，所以相比独立子查询较慢。

UNCACHEABLE SUBQUREY

无法使用缓存的子查询。对于依赖子查询，对于外部上下文中的变量的每一组不同值，子查询只重新计算一次。对于不可缓存的子查询，将为外部上下文的每一行都重新计算子查询。例如当使用了@@来引用系统变量的时候，不会使用缓存。

UNION

若第二个 SELECT 出现在 UNION 之后，则被标记为 UNION；若 UNION 包含在 FROM子句的子查询中，外层 SELECT 将被标记为：DERIVED。

在Stackoverflow上的提问

table

表明查询是基于哪张表的。

partitions

代表匹配的分区，需要结合表分区才可以看到。

type（重要）

查询的联结（join）类型，是较为重要的一个指标，结果值从最好到最坏依次是：

system > const > eq_ref > ref > fulltext > ref_or_null > index_merge > unique_subquery > index_subquery > range > index > all

一般来说得保证至少达到range级别，要求能达到 ref。

ref: 《阿里巴巴Java开发手册》

【推荐】 SQL性能优化的目标：至少要达到 range 级别，要求是 ref级别，如果可以是 consts 最好。

system

表只有一行记录（=system table），这是 const 类型的特列，平时不会出现，这个也可以忽略不计。

const

表示通过索引一次就找到了，const 用于比较 primary key 或者 unique 索引。因为只匹配一行数据，所以很快。如将主键置于 where 列表中，MySQL 就能将该查询转换为一个常量，也就是下面将t1看成常量表（const table）。所以const的两个特点就是：

  • 用于比较当前表格的primary key 或者 unique 索引，并且为 =；
  • 后面比较的是个常数。

eq_ref

唯一性索引扫描，对于每个索引键，当前表中只有一条记录与之匹配。常见于primary key 索引或unique not null索引扫描。所以eq_ref的两个特点就是：

  • 用于比较当前表格的 primary key 或者 unique not null 索引，并且为 =；
  • 后面比较的可以为一个表达式或者是其他表格在当前表格之前读取的列。

ref（最好到）

非唯一性索引扫描，返回匹配某个单独值的所有行。本质上也是一种索引访问，它返回所有匹配某个单独值的行，但它可能会找到多个符合条件的行，所以不是primary key或unique索引，是一般索引，应该属于查找和扫描的混合体。所以ref的两个特点是：

  • 用于比较当前表格的除了primary key 或者 unique 的索引，并且为 = 或 <=>；
  • 后面比较的可以为一个表达式或者是其他表格在当前表格之前读取的列。

注意，对于上面的情况并不总是如此，这时是因为t1 和 t2都只有一条记录，如果 t1 的记录比 t2 多，这个时候就会看到一个奇怪的现象，ref 会变成 index，我也是无意中才发现这个现象。

查阅了一些资料，也进行了一些思考。我觉得是因为 MySQL 对SQL语句进行了优化，按照小表驱动大表的原则，小表（外表）为t2只有一行，而且可以走索引，那进行hash join似乎性能更高，也可能不对，未来懂了再来更新吧。

fulltext

联结用到了全文索引。官方文档就这一句话，看来它并不重要。

ref_or_null

这个跟ref差不多，不同之处在于它对于包含NULL值的行多了一个额外的搜索。所以它的发生条件一般如下：

  • 用于比较当前表格的除了primary key 或者 unique 的索引，并且为 = 或 <=>；
  • 后面比较的可以为一个表达式或者是其他表格在当前表格之前读取的列，并且关联一个对NULL值判断的条件。

index_merge

顾名思义，这个表明在查询过程中多个索引组合使用，也就是使用到了索引合并优化，通常出现在有 or 关键字的 sql 中。

unique_subquery

子查询中的唯一索引。类似于这种：

index_subquery

该联接类型类似于 unique_subquery，区别就是子查询中一个是主键一个不是主键，利用索引来关联子查询，不再全表扫描。

range（至少到）

只检索给定范围的行，使用一个索引来选择行。key 列显示使用了哪个索引一般就是在你的 where 语句中出现了 between、<、>、in 等这种范围查询。它比全表扫描要好，因为它只需要开始于索引的某一点，而结束于另一点，不用扫描全部索引。所以它的出现一般：

  • 用于比较当前表格的任意索引；
  • 使用 =, <>, >, >=, <, <=, IS NULL, <=>, BETWEEN, LIKE, 或 IN() 操作符。

index

index 跟 ALL 类似，但它多了一步对索引树的扫描，出现 index 是 sql 使用了索引但是没用通过索引进行过滤，一般是使用了覆盖索引或者是利用索引进行了排序分组。它又可分为以下两种情况：

  • 如果索引是查询的覆盖索引，并且可以用来满足从表中需要的所有数据，那么只扫描索引树。此时 Extra 列显示Using index。仅index扫描通常比All扫描都快，因为索引的大小通常比表数据小；
  • 使用从索引读取数据以按索引顺序查找数据行来执行全表扫描。此时 Extra 列不会出现 Using index。

例如上面索引只包含id列，所以select id 为 index。

all

性能最差的，遍历全表以找到匹配的行。

possible_keys

显示可能应用在这张表中的索引，一个或多个。查询涉及到的字段上若存在索引，则该索引将被列出，但不一
定被查询实际使用。

key （重要）

这里才是实际用到的索引。如果为NULL，则表明没有使用索引。

key_len

表示索引中使用的字节数，可通过该列计算查询中使用的索引的长度。 key_len 字段能够帮助我们检查哪些索引被使用，ken_len 越长，说明索引使用的越充分。在估算哪些键被使用时，有以下注意的点：

  • 先看索引上字段的类型和长度，例如 int=4 ; varchar(20) = 20 ; char(20) = 20；
  • 如果是 varchar 或者 char 这种字符串字段，视字符集要乘不同的值，比如 utf-8 要乘 3, GBK 要乘 2，而现在所用的 utf8mb4 要乘 4；
  • varchar 这种动态字符串要加 2 个字节（有两个字节用于存储长度 ）；
  • 由于key的存储格式，允许为NULL的列比规定NOT NULL的列长一个字节。

从上面可以看到 key_len = 100 * 4(utf8mb4) + 2(长度字段) + 1(允许为NULL);

ref

显示索引的哪一列被使用了，也可能为常量，显示哪些列或常量被用于查找索引列上的值。例如上例将常量’1’用于查找索引列上的值。

rows （重要）

rows 列显示 MySQL 认为它执行查询时必须检查的行数，当然是越少越好，但注意在InnoDB中只是一个估计值，未必准确。这个值可以跟下面的 filtered 结合使用用于估计与下一个表联结的行数。

filtered

通过过滤条件的记录的估计百分比，最大值是100，这意味着没有对行进行筛选。rows 显示所需要检查的估计行数，rows × filtered 显示与下表进行联结的行数。例如，如果 rows 为 1000，过滤后为 50.00(50%)，则需要与下表连接的行数为1000 × 50% = 500。

Extra （重要）

其他的额外信息。别看它是额外信息，但它包含的信息却很重要而且也很多，我们挑重要的和经常见到的学习一下。

If you want to make your queries as fast as possible, look out for Extra column values of Using filesort and Using temporary, or, in JSON-formatted EXPLAIN output, for using_filesort and using_temporary_table properties equal to true.

Using filesort（糟糕）

说明 MySQL 会对数据使用一个外部的索引排序，而不是按照表内的索引顺序进行读取。MySQL 中无法利用索引完成的排序操作称为“文件排序”。

我们上面建立了c1 和 c2的联合索引，根据最左前缀原则，根据key_len我们可以判断查询用到了索引的前半部分，但排序order by却没有用到，所以在排序时 MySQL 会自己调用了“文件排序”，这会造成性能的下降。

ref: 《阿里巴巴Java开发手册》

【推荐】如果有 order by的场景，请注意利用索引的有序性。order by 最后的字段是组合 索引的一部分，并且放在索引组合顺序的最后，避免出现 file_sort的情况，影响查询性能。

正例：where a=? and b=? order by c; 索引：a_b_c

反例：索引中有范围查找，那么索引有序性无法利用，如：WHERE a>10 ORDER BY b; 索引 a_b无法排序。

Using temporary（非常糟糕）

使用了临时表保存中间结果，MySQL 在对查询结果排序时使用临时表。常见于排序 order by 和分组查询 group
by用不同方式列出，这个使用了临时表会更慢。

Using index（不错）

Using index 代表表示相应的 select 操作中使用了覆盖索引(Covering Index)，避免访问了表的数据行，效率不错！
如果同时出现 using where，表明索引被用来执行索引键值的查找；如果没有同时出现 using where，表明索引只是 用来读取数据而非利用索引执行查找，可能是利用索引进行了排序或分组。这个在上面的index那里有过详细说明。

第一个例子只用于读取，第二个进行了查找，这里in出现在最后，符合最左前缀原则，所以可以用覆盖索引。关于最左前缀原则，在后面索引那一节会详细讲解。

Using where

表明使用了where 过滤。

impossible where

where 子句的值总是 false，不能用来获取任何元组，这个时候要检查一下是不是写错了。

Using join buffer

用于联结表时的表联结优化算法使用的连接缓存，MySQL 8.0.20 之后默认（强制）都是 Using join buffer (hash join)，在MySQL中的联结那一节我们有过详细的介绍。

select tables optimized away

在没有 GROUP BY 子句的情况下，基于索引优化 MIN/MAX 操作或者对于 MyISAM 存储引擎优化 COUNT(*) 操作，不必等到执行阶段再进行计算，查询执行计划生成的阶段即完成优化。

ref: MySQL 8.0 Reference Manual
